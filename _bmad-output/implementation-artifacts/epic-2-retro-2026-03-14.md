## What Went Well

1. **Epic 1 retro action items fully applied** — All 3 process improvements from Epic 1's retro were implemented: Python SDK wire protocol analysis in every story, AWS SDK builder return type documentation, and architecture doc discrepancy tracking. This prevented implementation errors and eliminated trial-and-error.

2. **Pattern velocity acceleration** — Each story built on the previous one's patterns. Wait established START-only checkpoint + double-check. Callback added two-phase and `get_operation()`. Invoke reused both. By Story 2.3, code review passed with zero findings.

3. **Two-phase callback design** — The most complex story (2.2) introduced a novel two-phase API (`create_callback` + `callback_result`) matching the Python SDK. The design correctly separated the non-suspending registration from the potentially-suspending result check.

4. **QA integration tests** — Post-epic QA automation added `with_wait`, `with_callback`, `with_invoke` to MockDurableContext and 6 integration tests exercising multi-operation workflows including a full step→wait→callback→invoke→step sequence.

5. **Zero tech debt carried** — All findings resolved within the same review cycle. No items deferred.

## Challenges

1. **Rust generics-on-impl constraint** — Story 2.3's first attempt at a `handle_invoke_completed<T>` helper method failed because generic `T` can't be scoped to a helper on an `impl` block without making the whole impl generic. Quickly refactored to static methods (`deserialize_invoke_result<T>`, `extract_invoke_error`).

2. **Three distinct operation patterns** — Wait (server-completed, no result), callback (external-completed, two-phase), and invoke (server-completed, with payload+result) each needed different checkpoint/replay handling. Pattern differences weren't always obvious until implementation.

3. **track_replay semantics** — Callback's `callback_result` deliberately does NOT call `track_replay` (it's not a durable operation). This design decision needed explicit documentation to prevent confusion.

## Key Insights

1. **Operation taxonomy** — Four categories emerged: SDK-completed (step), server-completed-no-result (wait), server-completed-with-result (invoke), external-completed-two-phase (callback). Understanding the category upfront simplifies implementation.

2. **`get_operation()` was the right abstraction** — Added in Story 2.2 for callbacks (needs any-status lookup), reused in 2.3 for invoke's STARTED/PENDING check. Will be essential for Epic 3.

3. **Code review ROI curve continues** — Epic 1 noted diminishing returns on later stories. Epic 2 confirms: Story 2.1 (first new operation type) was clean, 2.2 (most complex) had 2 minor doc issues, 2.3 (pattern-following) was a clean pass. Reviews remain valuable for novel patterns but routine for pattern-followers.
