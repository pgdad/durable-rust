## Story

As a developer,
I want to define durable Lambda handlers using a `#[durable_execution]` attribute macro,
So that I can write handlers with minimal boilerplate in the most idiomatic Rust style (mirroring #[tokio::main]).

## Acceptance Criteria

1. **Given** the durable-lambda-macro crate **When** I annotate an async function with `#[durable_execution]` **Then** the macro generates lambda_runtime handler registration and DurableContext setup (FR32) **And** the function receives a context with access to all 8 core operations

2. **Given** the proc-macro crate **When** I examine the structure **Then** it contains lib.rs (attribute macro entry point with `proc-macro = true`) and expand.rs (code generation logic) **And** it depends on syn, quote, and proc-macro2 for macro expansion

3. **Given** the generated code from the macro **When** I run `cargo expand` on a handler using `#[durable_execution]` **Then** the expanded code is readable and debuggable **And** it produces the same runtime behavior as the closure-native approach

4. **Given** the macro-generated handler **When** a developer uses it with AI coding tools **Then** the attribute macro pattern is reliably generated by Claude Code and Copilot (NFR13)

5. **Given** the durable-lambda-macro crate **When** I examine its dependencies **Then** it depends only on proc-macro tooling (syn, quote, proc-macro2) — NOT on durable-lambda-core at compile time (NFR8)

6. **Given** all public types, traits, and methods added in this story **When** I run `cargo test --workspace` **Then** all tests pass including new macro tests **And** all doc tests compile

## Dev Notes

### Architecture: Proc-Macro Crate Isolation

Proc-macro crates (`proc-macro = true`) have special constraints:
- They run at **compile time** in the compiler's process
- They can only export procedural macros — no regular types, structs, or functions
- They CANNOT depend on runtime crates they generate code for (circular dependency)

This means `durable-lambda-macro` does NOT depend on `durable-lambda-core`. Instead:
- The macro generates code that **references** `durable_lambda_core::*` paths
- The **user's crate** must depend on both `durable-lambda-macro` (for the attribute) and `durable-lambda-core` (for the types)
- This differs from the "single dependency" pattern of closure/trait/builder approach crates

### Architecture Doc Verified

The architecture doc says the macro crate "depends on durable-lambda-core" but this is **architecturally incorrect** for proc-macro crates. The current Cargo.toml correctly has only syn/quote/proc-macro2 as dependencies. The generated code references `durable_lambda_core` paths that the user's crate resolves.

**Discrepancy documented:** Architecture doc section on proc-macro dependencies is inaccurate — proc-macro crates cannot have runtime dependencies on the crates they generate code for.

### What the Macro Generates

User writes:
```rust
use durable_lambda_macro::durable_execution;
use durable_lambda_core::context::DurableContext;
use durable_lambda_core::error::DurableError;

#[durable_execution]
async fn handler(event: serde_json::Value, mut ctx: DurableContext) -> Result<serde_json::Value, DurableError> {
    let result: Result<i32, String> = ctx.step("validate", || async { Ok(42) }).await?;
    Ok(serde_json::json!({"result": result.unwrap()}))
}
```

Macro generates (conceptually):
```rust
// Original handler function preserved
async fn handler(event: serde_json::Value, mut ctx: DurableContext) -> Result<serde_json::Value, DurableError> {
    let result: Result<i32, String> = ctx.step("validate", || async { Ok(42) }).await?;
    Ok(serde_json::json!({"result": result.unwrap()}))
}

// Generated main() function
#[tokio::main]
async fn main() -> Result<(), lambda_runtime::Error> {
    let config = aws_config::load_defaults(aws_config::BehaviorVersion::latest()).await;
    let client = aws_sdk_lambda::Client::new(&config);
    let backend = std::sync::Arc::new(durable_lambda_core::backend::RealBackend::new(client));

    lambda_runtime::run(lambda_runtime::service_fn(|event: lambda_runtime::LambdaEvent<serde_json::Value>| {
        let backend = backend.clone();
        async move {
            let (payload, _lambda_ctx) = event.into_parts();
            // ... parse DurableExecutionArn, CheckpointToken, operations ...
            let durable_ctx = durable_lambda_core::context::DurableContext::new(
                backend, arn, token, operations, next_marker
            ).await.map_err(|e| Box::new(e) as Box<dyn std::error::Error + Send + Sync>)?;
            let user_event = extract_user_event(&initial_state);
            let result = handler(user_event, durable_ctx).await
                .map_err(|e| Box::new(e) as Box<dyn std::error::Error + Send + Sync>)?;
            Ok::<serde_json::Value, Box<dyn std::error::Error + Send + Sync>>(result)
        }
    })).await
}
```

### Reference Implementation: Closure Crate's `run()` Function

The macro expansion must replicate the same runtime behavior as `durable_lambda_closure::run()`. The key logic to replicate from `crates/durable-lambda-closure/src/handler.rs`:

1. AWS config + Lambda client + RealBackend setup
2. `lambda_runtime::run(service_fn(...))` registration
3. Event parsing: `DurableExecutionArn`, `CheckpointToken`, `InitialExecutionState`
4. `parse_operations()` — builds `Operation` objects from JSON
5. `parse_operation_type()` — string to `OperationType` enum conversion
6. `parse_operation_status()` — string to `OperationStatus` enum conversion
7. `extract_user_event()` — extracts user's original event from first EXECUTION operation
8. `DurableContext::new()` with pagination
9. Handler invocation with user event + context

### Key Difference from Closure Approach

In the closure approach, the user calls `durable_lambda_closure::run(handler).await` from their own `main()`. In the macro approach, the macro **generates** the `main()` function. The user's annotated function becomes the handler, and the macro creates all the boilerplate around it.

This means the user writes **less code** — no `#[tokio::main]`, no `main()` function, just the handler with `#[durable_execution]`.

### Note on DurableContext vs ClosureContext

Unlike the closure approach which wraps `DurableContext` in `ClosureContext`, the macro approach passes `DurableContext` directly to the user's handler. This is because:
- The macro crate can't define wrapper types (proc-macro crates only export macros)
- `DurableContext` already has all 8 operations as public methods
- The user imports `DurableContext` from `durable_lambda_core` directly

### Testing Proc-Macros

Proc-macro testing approaches:
1. **`trybuild` crate** — compile-time test harness that verifies macro expansion compiles (or fails with expected errors). Good for testing error messages.
2. **Manual token inspection** — call the expansion function directly and compare output tokens
3. **Integration tests** — write actual handlers with `#[durable_execution]` and verify they compile

Recommended: Use `trybuild` for compile-pass/compile-fail tests. Add it as a dev-dependency.

### Helper Functions Strategy

The `parse_operations()`, `parse_operation_type()`, `parse_operation_status()`, and `extract_user_event()` helper functions from the closure crate need to be available to the generated code. Options:

1. **Inline in generated code** — generates verbose but self-contained code
2. **Move helpers to core crate** — add a `durable_lambda_core::event` module with these helpers, then generated code calls `durable_lambda_core::event::parse_operations()`
3. **Duplicate in macro expansion** — generate the helpers as local functions in main()

**Recommended: Option 2** — move the helpers to `durable_lambda_core` so they're shared across all approach crates. This prevents code duplication and makes the generated code cleaner. The trait and builder crates will also need these helpers.
