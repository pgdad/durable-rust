5. **TEST-02: test_serialization_type_mismatch_returns_deserialization_error** — Use `OperationIdGenerator` to get op_id. Build an `Operation` with `OperationStatus::Succeeded` and `StepDetails` containing `result(r#"true"#)` (a boolean JSON). Call `ctx.step::<i32, String, _, _>("test", ...)` which expects i32 but the cached data is boolean. Assert the result is `Err(DurableError::Deserialization { .. })`.

6. **TEST-03: test_checkpoint_failure_propagates** — Use the `FailingMockBackend` (not MockDurableContext). Build a `DurableContext::new(Arc::new(FailingMockBackend::new()), ...)` with empty history (execute mode). Call `ctx.step("test", || async { Ok(42i32) })`. The step will try to checkpoint START and get an error. Assert the result is `Err(DurableError::CheckpointFailed { .. })`.

7. **TEST-04: test_retry_exhaustion_surfaces_user_error** — Use `OperationIdGenerator` to get op_id. Build an `Operation` with `OperationStatus::Pending` and `StepDetails::builder().attempt(4).build()` (attempt 4 = retries exhausted when retries(3)). Use a regular `MockDurableContext::new().build()` but construct the context directly with `DurableContext::new(...)` passing the pre-loaded op. Use a working mock backend (from step.rs test pattern). Call `ctx.step_with_options("test", StepOptions::new().retries(3), || async { Err("final failure".to_string()) })`. Assert the result is `Ok(Err("final failure"))` (NOT `Err(StepRetryScheduled)`). Key: `attempt(4)` ensures `4 <= 3` is false, so retries are exhausted.

8. **TEST-05: test_callback_timeout_returns_callback_failed** — Use `OperationIdGenerator` to get op_id. Build a callback `Operation` with `OperationType::Callback`, `OperationStatus::TimedOut`, and `CallbackDetails::builder().callback_id("cb-timeout-1").build()`. Construct context with this operation. Call `ctx.create_callback("approval", CallbackOptions::new()).await.unwrap()` then `ctx.callback_result::<String>(&handle).await`. Assert the error is `DurableError::CallbackFailed { .. }`.

9. **TEST-06: test_callback_explicit_failure_returns_callback_failed** — Same as TEST-05 but use `OperationStatus::Failed` and add an `ErrorObject` to the callback details (via `ContextDetails` or directly on the operation, depending on the callback operation structure). Assert the error message contains the failure reason.

10. **TEST-07: test_invoke_error_returns_invoke_failed** — Use `OperationIdGenerator` to get op_id. Build a `ChainedInvoke` `Operation` with `OperationType::ChainedInvoke`, `OperationStatus::Failed`, and `ChainedInvokeDetails` with error info. Construct context. Call `ctx.invoke(...)`. Assert `Err(DurableError::InvokeFailed { .. })`.
