  <action>
Append 8 assertion helper functions to the end of `scripts/test-helpers.sh`. Each helper takes a binary name, invokes it via `invoke_sync`, and asserts 2-3 key response fields using `jq -r`. Follow the exact Phase 16 pattern: `get_alias_arn -> invoke_sync -> IFS='|' read -r status fn_error _ response_body -> assertions -> success echo`.

**Per user decision:** The durable execution service unwraps SUCCEEDED responses, so all assertions check raw user JSON (no Status envelope). All handlers return HTTP 200 with no FunctionError.

The 8 helpers and their assertions:

1. `assert_basic_steps(binary)` — Payload: `{"order_id":"test-order-001"}`. Assert: `order_id == "test-order-001"` (round-trip), `details.status == "found"`. Echo: "basic steps executed and replayed correctly via $binary".

2. `assert_step_retries(binary)` — Payload: `{}`. Assert: `result.api_response == "success"`. Echo: "step retries completed successfully via $binary".

3. `assert_typed_errors(binary)` — Per user decision: test BOTH paths in one call.
   - Success path: Payload `{"amount":50}`. Assert: HTTP 200, no FunctionError, `transaction_id == "txn_50"`.
   - Error path: Payload `{"amount":2000}`. Assert: HTTP 200, no FunctionError (domain error, not Lambda error), `error == "insufficient_funds"`.
   - Echo: "typed error correctly serialized through durable execution via $binary".

4. `assert_parallel(binary)` — Payload: `{}`. Assert: `parallel_results | length == 3`. Per research open question about parallel order non-determinism, use `jq '[.parallel_results[] | .branch] | sort | join(",")'` and assert result is `"a,b,c"` (sorted membership check, not index access). Echo: "parallel fan-out completed with 3 branches via $binary".

5. `assert_map(binary)` — Payload: `{}`. Assert: `processed_orders | length == 4`, `processed_orders[0].status == "done"` (map preserves input order). Echo: "map operation processed 4 orders via $binary".

6. `assert_child_contexts(binary)` — Payload: `{}`. Assert: `child_result.validation == "passed"`, `parent_result == "parent_validation"`. Echo: "child context isolated namespace, parent step ran independently via $binary".

7. `assert_replay_safe_logging(binary)` — Payload: `{"order_id":"test-order-001"}`. Per user decision: response-only validation, no CloudWatch queries. Assert: `order_id == "test-order-001"`, `result.processed == "true"`. Echo: "replay-safe logging handler executed without duplicate log side effects via $binary".

8. `assert_combined_workflow(binary)` — Payload: `{"order_id":"test-order-001","total":99.99}`. Per user decision: validate order_id, payment.charged, fulfillment non-null, post_processing non-null. Assert: `order_id == "test-order-001"`, `payment.charged == "true"`, `fulfillment != "null"`, `post_processing != "null"`. Add a comment noting this invocation blocks ~35+ seconds due to ctx.wait(30s). Echo: "combined workflow completed with payment, fulfillment, and post-processing via $binary".

**Critical patterns to follow:**
- Every helper starts with `local binary="$1"` then `local fn_arn; fn_arn=$(get_alias_arn "$binary")`
- Parse invoke_sync output with `IFS='|' read -r status fn_error _ response_body <<< "$result"` (the `_` consumes exec_arn)
- Every helper checks `[[ "$status" == "200" ]]` and `[[ -z "$fn_error" ]]` first
- Error messages include `body=$response_body` for debuggability
- Use `{ echo "message"; return 1; }` for failures (matching Phase 16 pattern)

**Do NOT:**
- Add a shebang (file is sourced)
- Duplicate the invoke_sync or get_alias_arn logic
- Check for Status envelope fields (.Status, .Result)
- Use index access for parallel branch assertions (use sorted membership)
