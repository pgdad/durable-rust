  <action>
In each of the 7 operation files, add a `tracing::info_span!` immediately after `generate_operation_id()` and before any replay check or checkpoint call. The pattern is identical for all 7 — only `op.type` differs.

Pattern to insert in each file:

```rust
let span = tracing::info_span!(
    "durable_operation",
    op.name = name,
    op.type = "{TYPE}",
    op.id = %op_id,
);
let _guard = span.enter();
```

Where `{TYPE}` is the operation-specific type string:
- `step.rs` (in `step_with_options`): `op.type = "step"`
- `wait.rs` (in `wait`): `op.type = "wait"`
- `callback.rs` (in `create_callback`): `op.type = "callback"`
- `invoke.rs` (in `invoke`): `op.type = "invoke"`
- `parallel.rs` (in `parallel`): `op.type = "parallel"`
- `map.rs` (in `map`): `op.type = "map"`
- `child_context.rs` (in `child_context`): `op.type = "child_context"`

IMPORTANT notes:
- Use `span.enter()` guard pattern (not `.instrument()`) per research recommendation. These are `&mut self` methods where moving the body into an async block for `.instrument()` is impractical due to the mutable borrow.
- The guard is created at the top and lives for the rest of the function. Clippy may warn about holding the guard across `.await` — this is acceptable here because: (a) these methods run on a single-threaded test runtime in tests, and (b) for FEAT-17-20 the requirement is that spans are emitted with correct fields, which `tracing-test` verifies regardless of guard lifetime.
- If Clippy does flag `clippy::await_holding_enter_guard`, add `#[allow(clippy::await_holding_enter_guard)]` on the function.
- `op.type` is a Rust keyword — in `tracing::info_span!` field names, use `op.type` literally (the dot makes it a structured field name, not a Rust identifier).
- No new `use` statements needed at the module level — `tracing::info_span!` is a macro that resolves without importing.

For `callback.rs`: The span goes in `create_callback()` only (the active operation that sends a checkpoint). `callback_result()` is a synchronous replay-only method and does not need a span.

For `parallel.rs` and `map.rs`: The span wraps the outer operation. Per research, spawned branches inheriting the parent span is a secondary concern for FEAT-18 and NOT required here. The outer span covering the parallel/map block is sufficient. FEAT-18 specifically requires "parent-child span hierarchy matches context nesting" which is satisfied by `child_context` nesting (where the child's operations run synchronously within the parent's span guard).
