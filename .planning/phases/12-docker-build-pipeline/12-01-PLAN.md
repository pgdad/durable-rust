**Step 2: Rewrite `examples/Dockerfile`** with the cargo-chef three-stage pattern. The exact structure:

Stage 0 — `chef` base: `FROM lukemathwalker/cargo-chef:latest-rust-1 AS chef` with `WORKDIR /usr/src/app`.

Stage 1 — `planner`: `FROM chef AS planner`, `COPY . .`, `RUN cargo chef prepare --recipe-path recipe.json`. This generates a minimal recipe.json containing only dependency metadata (not source code), so it changes only when Cargo.toml/Cargo.lock change.

Stage 2 — `builder`: `FROM chef AS builder`, `ARG PACKAGE=closure-style-example`, `COPY --from=planner /usr/src/app/recipe.json recipe.json`, `RUN cargo chef cook --release --recipe-path recipe.json` (full workspace cook, NOT per-crate — see research open question 1 about cross-crate deps), then `COPY . .`, `RUN cargo build --release -p "${PACKAGE}"`. The cook line is the cached dependency layer; the build line compiles only the specified crate.

Stage 3 — runtime: `FROM public.ecr.aws/lambda/provided:al2023`, `ARG BINARY_NAME=closure-basic-steps`, `COPY --from=builder "/usr/src/app/target/release/${BINARY_NAME}" "${LAMBDA_RUNTIME_DIR}/bootstrap"`, `CMD ["handler"]`.

Key details:
- `cargo chef cook` uses full workspace scope (no `-p` flag) per research recommendation — avoids cross-crate dependency resolution issues since all 4 example crates depend on durable-lambda-core
- `PACKAGE` drives `cargo build -p` (which crate to compile)
- `BINARY_NAME` drives `COPY` in runtime stage (which binary to extract)
- Both have defaults for standalone use but are always overridden by build-images.sh
- Preserve the existing file-level doc comment explaining usage, updated for the new ARGs
