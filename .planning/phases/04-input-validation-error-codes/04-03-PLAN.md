<objective>
Replace all 13 silent `if let Some(token)` checkpoint token handling sites with defensive `.ok_or_else(...)` error propagation, so a missing checkpoint_token in AWS responses surfaces as a typed DurableError instead of silently continuing with a stale token. Add a test proving the None token error path works.

Purpose: A missing checkpoint_token after a checkpoint write indicates an AWS API contract violation. Silent continuation with a stale token causes subtle downstream corruption. Surfacing it as DurableError::checkpoint_failed makes the failure visible and debuggable.
Output: All 7 operation files updated with defensive checkpoint token handling across 13 sites, plus a test exercising the None token path.
</objective>

#[tokio::test]
async fn checkpoint_none_token_returns_error() {
    let backend = Arc::new(NoneTokenMockBackend);

    let mut ctx = DurableContext::new(
        backend,
        "arn:test".to_string(),
        "initial-token".to_string(),
        Vec::new(), // empty history = executing mode
    );

    // Attempt a step -- the first checkpoint (START) will return None token
    let result: Result<i32, DurableError> = ctx
        .step("test_step", || async { Ok(42) })
        .await;

    // Must be an error, not a silent success with stale token
    let err = result.expect_err(
        "step should fail when checkpoint response has None checkpoint_token"
    );

    // Verify it's specifically a CheckpointFailed error
    match &err {
        DurableError::CheckpointFailed { operation_name, .. } => {
            assert!(
                operation_name.contains("test_step"),
                "error should reference the operation name, got: {}",
                operation_name
            );
        }
        other => panic!(
            "expected DurableError::CheckpointFailed, got: {:?}",
            other

        "error message should mention missing checkpoint_token, got: {}",
        err_msg
    );
}
