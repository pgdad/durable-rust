**TEST-19: test_deterministic_replay_100_runs**

Run the same replay scenario 100 times. Each iteration:
1. Build `MockDurableContext::new().with_step_result("step_a", r#"42"#).with_step_result("step_b", r#""hello""#).build().await`
2. Call `ctx.step("step_a", ...)` and `ctx.step("step_b", ...)` with panicking closures (should never execute in replay)
3. Assert results are `42` and `"hello"` respectively
4. Assert zero checkpoints: `calls.lock().await.len() == 0`

All 100 iterations must produce identical results. The test proves OperationIdGenerator is deterministic and ReplayEngine produces consistent results.

```rust
#[tokio::test]
async fn test_deterministic_replay_100_runs() {
    for i in 0..100 {
        let (mut ctx, calls, _ops) = MockDurableContext::new()
            .with_step_result("step_a", r#"42"#)
            .with_step_result("step_b", r#""hello""#)
            .build()
            .await;

        let r1: Result<i32, String> = ctx
            .step("step_a", || async { panic!("should not execute in replay") })
            .await
            .unwrap();
        let r2: Result<String, String> = ctx
            .step("step_b", || async { panic!("should not execute in replay") })
            .await
            .unwrap();

        assert_eq!(r1.unwrap(), 42, "iteration {i}: step_a result mismatch");
        assert_eq!(r2.unwrap(), "hello", "iteration {i}: step_b result mismatch");

        let captured = calls.lock().await;
        assert_eq!(captured.len(), 0, "iteration {i}: replay should produce 0 checkpoints");
    }
}
```

**TEST-20: test_duplicate_operation_ids_last_writer_wins**

Construct two Operations with the SAME operation ID but different result values. Pass both to `DurableContext::new()` — the second should win (HashMap last-writer-wins).

1. Use `OperationIdGenerator::new(None)` to compute the first operation ID (counter=1)
2. Build `op_first` with that ID and result `r#"999"#`
3. Build `op_second` with the SAME ID and result `r#"42"#`
4. Create `MockBackend::new("mock-token")` to get `(backend, calls, ops)`
5. Call `DurableContext::new(Arc::new(backend), "arn:test".into(), "tok".into(), vec![op_first, op_second], None).await.unwrap()`
6. Call `ctx.step("any", || async { panic!("not executed") }).await.unwrap()`
7. Assert the result is `42` (second op wins), NOT `999`

```rust
use durable_lambda_core::operation_id::OperationIdGenerator;
use durable_lambda_testing::mock_backend::MockBackend;
// ... other imports

#[tokio::test]
async fn test_duplicate_operation_ids_last_writer_wins() {
    let mut gen = OperationIdGenerator::new(None);
    let op_id = gen.next_id();

    let op_first = Operation::builder()
        .id(&op_id)
        .r#type(OperationType::Step)
        .status(OperationStatus::Succeeded)
        .start_timestamp(DateTime::from_secs(0))
        .step_details(StepDetails::builder().result(r#"999"#).build())
        .build()
        .unwrap();

    let op_second = Operation::builder()
        .id(&op_id)
        .r#type(OperationType::Step)
        .status(OperationStatus::Succeeded)
        .start_timestamp(DateTime::from_secs(0))
        .step_details(StepDetails::builder().result(r#"42"#).build())
        .build()
        .unwrap();

    let (backend, _calls, _ops) = MockBackend::new("mock-token");
    let mut ctx = DurableContext::new(
        Arc::new(backend),
        "arn:test".into(),
        "tok".into(),
        vec![op_first, op_second],
        None,
    )
    .await
    .unwrap();

    let result: Result<i32, String> = ctx
        .step("any", || async { panic!("not executed in replay") })
        .await
        .unwrap();
    assert_eq!(result.unwrap(), 42, "second op (last writer) should win");
}
```

**TEST-21: test_history_gap_triggers_execute_path**

Create history with op at counter=1 (Succeeded) but NO op at counter=2, then op at counter=3 (Succeeded). When the handler calls step 1 (replay), step 2 (execute — gap), step 3 — step 2 should hit the execute path because its ID is missing from history.

1. Use `OperationIdGenerator::new(None)` to compute IDs for counters 1, 2, 3
2. Build operations for counter=1 (result `r#"1"#`) and counter=3 (result `r#"3"#`) — skip counter=2
3. Create `MockBackend::new("mock-token")` and `DurableContext::new(...)` with ops for 1 and 3
4. Call step 1: should replay (result 1, no checkpoint)
5. Call step 2: should EXECUTE (gap — ID for counter=2 not in history). The closure runs and returns a value. Checkpoint IS made.
6. NOTE: Step 3 may not replay from pre-loaded history because after step 2 executes, the replay engine transitions to Executing mode (all completed ops visited). This is acceptable — document this as defined behavior.

```rust
#[tokio::test]
async fn test_history_gap_triggers_execute_path() {
    let mut gen = OperationIdGenerator::new(None);
    let id1 = gen.next_id(); // counter=1
    let _id2 = gen.next_id(); // counter=2 — intentionally skipped in history
    // Note: we don't need id3 for the assertion; we just need to verify step2 executes

    let op1 = Operation::builder()
        .id(&id1)
        .r#type(OperationType::Step)
        .status(OperationStatus::Succeeded)
        .start_timestamp(DateTime::from_secs(0))
        .step_details(StepDetails::builder().result(r#"1"#).build())
        .build()
        .unwrap();

    // No op for id2 — this is the gap

    let (backend, calls, _ops) = MockBackend::new("mock-token");
    let mut ctx = DurableContext::new(
        Arc::new(backend),
        "arn:test".into(),
        "mock-checkpoint-token".into(),
        vec![op1],  // Only op1 in history
        None,
    )
    .await
    .unwrap();

    // Step 1: replay path — op1 exists in history
    let r1: Result<i32, String> = ctx
        .step("step1", || async { panic!("not executed in replay") })
        .await
        .unwrap();
    assert_eq!(r1.unwrap(), 1);

    // Step 2: execute path — op2 NOT in history (gap)
    let r2: Result<i32, String> = ctx
        .step("step2", || async { Ok(222) })
        .await
        .unwrap();
    assert_eq!(r2.unwrap(), 222, "step2 closure should execute because its ID is missing from history");

    // Verify step2 made checkpoints (execute path) while step1 did not
    let captured = calls.lock().await;
    assert!(
        captured.len() >= 2,
        "step2 should have produced checkpoint calls (START + SUCCEED), got {}",
        captured.len()
    );
}
```

**test_checkpoint_token_evolution** — Verify that the checkpoint token passed to each checkpoint call matches the token returned by the previous checkpoint response.

Use `MockDurableContext::new().build().await` (execute mode, no pre-loaded ops). Run 2 steps. Examine the `CheckpointCall` records to verify token evolution.

The MockBackend initial token is "mock-checkpoint-token" (set during `MockDurableContext::build()`). After each checkpoint, MockBackend returns "mock-token" as the new token. So the evolution pattern is:
- Step 1 START: uses initial token "mock-checkpoint-token"
- Step 1 SUCCEED: uses "mock-token" (returned from Step 1 START response)
- Step 2 START: uses "mock-token" (returned from Step 1 SUCCEED response)
- Step 2 SUCCEED: uses "mock-token" (returned from Step 2 START response)

```rust
#[tokio::test]
async fn test_checkpoint_token_evolution() {
    let (mut ctx, calls, _ops) = MockDurableContext::new().build().await;

    // Execute two steps — each produces START + SUCCEED checkpoints
    let _: Result<i32, String> = ctx.step("s1", || async { Ok(1) }).await.unwrap();
    let _: Result<i32, String> = ctx.step("s2", || async { Ok(2) }).await.unwrap();

    let captured = calls.lock().await;

    // Should have 4 checkpoint calls: s1-START, s1-SUCCEED, s2-START, s2-SUCCEED
    assert_eq!(captured.len(), 4, "expected 4 checkpoint calls, got {}", captured.len());

    // First checkpoint (s1 START) uses the initial token
    assert_eq!(
        captured[0].checkpoint_token, "mock-checkpoint-token",
        "first checkpoint should use initial token"
    );

    // Subsequent checkpoints use the token returned by MockBackend ("mock-token")
    for (i, call) in captured.iter().enumerate().skip(1) {
        assert_eq!(
            call.checkpoint_token, "mock-token",
            "checkpoint {i} should use updated token from previous response"
        );
    }
}
```

**Key observation:** MockBackend always returns the same "mock-token" for every response. In production, each response returns a different token. This test still validates the evolution mechanism: the context correctly updates its token after each checkpoint call and uses the updated token for the next call. The initial -> updated transition (index 0 vs index 1+) proves the mechanism works.
