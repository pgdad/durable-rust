## BMAD Artifact Inventory — Complete Removal

### BMAD framework tooling — 508 files, 5 subdirectories (removed)

BMAD tooling framework that was installed into the project repository. Contained the full BMAD agent system, workflows, templates, and configuration. Tooling infrastructure only, not project artifacts.

Structure: `_config/` (manifests, IDE configs), `_memory/` (runtime memory), `bmm/` (agents), `core/` (core framework), `tea/` (tea module).

**Disposition:** Full directory removal. No content maps to GSD — GSD tooling lives at `~/.claude/get-shit-done/`, not in the project repository.

### BMAD output artifacts — 37 files, 5 subdirectories (removed)

Project artifacts generated by BMAD agents during v1.0 development. Historical record of how v1.0 was planned and built.

Structure: `brainstorming/` (session notes), `implementation-artifacts/` (31 files: story guides, retros, sprint status), `planning-artifacts/` (architecture, epics, PRD, product brief, readiness report), `test-artifacts/` (empty), `project-context.md`.

**Disposition:** Full directory removal. The v1.0 work these documents describe is already captured as `Validated` requirements in `.planning/PROJECT.md`. GSD tracks history through `MILESTONES.md`, git log, and `PROJECT.md`'s Validated section.

## Alternatives Considered

| Recommended | Alternative | When to Use Alternative |
|-------------|-------------|-------------------------|
| Full removal of BMAD tooling and output in separate commits | Archive to a branch before deletion | If team needs to reference BMAD artifacts post-transition — not needed here since v1.0 work is captured in PROJECT.md |

## What NOT to Use

| Avoid | Why | Use Instead |
|-------|-----|-------------|
| Migrating BMAD output content into `.planning/` | BMAD story/epic format doesn't map to GSD phase/plan format; creates noise without value | Capture summary in `MILESTONES.md` v1.0 entry; full history lives in git |
| Keeping BMAD tooling for reference | 508 files of framework tooling with no runtime value in a GSD workflow | Delete entirely; GSD tooling is at `~/.claude/get-shit-done/` |
| Expanding `config.json` unnecessarily | Minimal config is valid; GSD uses defaults | Leave minimal config unless specific behavior overrides are needed |

## Execution Order for This Milestone

Based on GSD `new-milestone` workflow:

1. **Research complete** — this file (STACK.md) + SUMMARY.md
2. **REQUIREMENTS.md creation** — single requirement: stand up GSD infra, remove BMAD
3. **MILESTONES.md creation** — log v1.0 retroactively to establish baseline
4. **ROADMAP.md creation** — 2-phase plan: (a) create remaining GSD files, (b) remove BMAD artifacts
5. **Execution** — create files, delete BMAD framework tooling and output artifacts, commit

The two phases should be separate commits per PROJECT.md Key Decisions: "Remove BMAD artifacts in separate commit."
